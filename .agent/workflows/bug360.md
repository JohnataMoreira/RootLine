# /bug360 — PROTOCOLO DE ERRO 360° + OPS_KNOWLEDGE_BASE (OBRIGATÓRIO)

## 🚨 PROTOCOLO DE ERRO 360° (REGRA DE OURO)

**Se surgir qualquer erro (Build, Deploy, Runtime ou Terminal), você DEVE seguir esta ordem rigorosa ANTES de propor qualquer correção:**

### Fase 1: Coleta de Evidência (O "Pente Fino")
1.  **Contexto:** Juntar o que estava sendo feito + o que foi implementado/corrigido por último.
2.  **Logs:** Executar `docker logs [container_name]` ou verificar logs do sistema.
3.  **Ambiente:** Verificar portas, permissões e variáveis de ambiente.

### Fase 2: Pesquisa de Elite (Deep Research)
**OBRIGATÓRIO:** Antes de propor qualquer correção, você DEVE realizar uma busca aprimorada:
-   **Documentação Oficial:** Pesquisar no site oficial da ferramenta envolvida (Ex: Dokploy Docs, Docker Docs, NestJS Docs).
-   **Comunidade e Fóruns:** Buscar no StackOverflow, GitHub Issues e Reddit pelo erro literal.
-   **Google:** Cruzar informações para encontrar a solução mais estável e Moderna.

### Fase 3: Proposta e Execução
-   Apresentar o diagnóstico baseado na pesquisa.
-   Propor a correção com plano de rollback.
-   Executar apenas após validação do contexto.

## 📚 OPS_KNOWLEDGE_BASE (Biblioteca de Erros)

Para evitar perder horas em erros recorrentes, você deve manter e consultar a pasta:
`[PROJETO]/.agent/knowledge/error_logs/`

**REGRA:** Sempre consulte esta pasta antes de iniciar um debug complexo. Cada erro resolvido deve ser documentado no formato:
-   **Erro:** Descrição literal.
-   **Contexto:** O que estava sendo feito.
-   **Causa Raiz:** O que realmente causou o problema.
-   **Solução:** O comando ou alteração exata.
-   **Prevenção:** Como evitar que ocorra novamente.
